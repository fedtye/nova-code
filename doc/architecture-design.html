v1.0 Architecture Design

Nova Agent CLI 架构设计

基于产品迭代路线图，定义 Nova Agent 从 MVP 核心到完整版的演进式架构：系统分层、模块职责、核心数据流、关键接口与架构决策。 遵循"从内核到外围、从单 Agent 到多 Agent、从基础能力到高级进化"的分阶段激活策略，技术栈统一采用 React + Node.js / TypeScript。

架构版本 v1.0 文档日期 2026/05/21 状态 已确认

01 架构总览

Nova Agent CLI 采用插件化分层架构，以 Node.js 单进程为运行时基座，通过事件总线驱动各层协作。 整体分为五层：表现层负责多模态用户交互，应用层管理会话与任务生命周期，智能层承载 Agent 核心能力（规划、编排、进化、记忆）， 扩展层提供 MCP、Hooks、Skill 三重开放体系，基础设施层提供文件、安全、存储、网络等底层能力。

迭代演进映射

架构组件按产品路线图分 5 个 Phase 渐进激活，前期严格遵循收敛版哲学（高信任、单 Agent、Human-in-the-loop），后期逐步开放完整版高级能力。

P1

Phase 1 — MVP 核心（v0.1~v0.4）

激活表现层（CLI/REPL/渲染）、应用层（会话/权限）、基础设施层（FS/沙箱）。模型层仅接入单一内置 Provider，所有修改需确认。

P2

Phase 2 — 能力扩展（v0.5~v0.8）

激活智能层（DAG 规划）与扩展层（MCP Client、Hooks Engine）。模型网关完成接口预留，管道模式与批量处理接入表现层。

P3

Phase 3 — 体验增强（v0.9~v0.11）

激活表现层 Dashboard、智能层上下文增强（@引用/记忆/压缩）、代码审查引擎。持久化记忆跨会话，Web 面板实时同步。

P4

Phase 4 — 高级智能（v0.12~v0.13）

激活智能层多 Agent 编排（并行 Specialist、消息总线）与GEPA 自进化循环。Skill 系统开始积累领域能力。

P5

Phase 5 — 产品成熟（v1.0）

激活消息网关（Slack/Telegram/Discord Bot）、扩展层市场体系（Skill Market + MCP Market）、企业版安全合规与私有化部署。

组件全景图

[User Input] ↓ Command Parser / REPL Engine ←→ Stream Renderer ↓ Session Manager ←→ Permission Controller ↓ Task Scheduler (DAG Queue + State Machine) ↓ Task Planner ←→ Model Gateway ↓ (sub-task DAG) Agent Orchestrator (并行 Specialist Agents) ↓ Context Manager ←→ RAG / Knowledge Graph / Memory Store ↓ MCP Client / Hooks Engine / Skill Repository ←→ Security Sandbox / FS Abstraction ↓ GEPA Loop (Generate → Evaluate → Prune → Archive) ↓ [Output + Memory Archive]

02 设计原则

🏠

本地优先 Local-First

核心逻辑、文件操作、模型推理全部在本地 Node.js 进程完成，数据不出本地，保障隐私与零延迟。

🔒

安全优先 Security-First

所有外部命令、网络请求、文件修改均经过沙箱审查与分级授权，默认拒绝高危操作，支持一键回滚。

🔌

开放扩展 Open Extension

MCP + Hooks + Skill 三重扩展体系，每层均定义标准化接口，社区可无缝接入新工具、新事件、新能力。

🔄

模型无关 Model-Agnostic（架构就绪）

模型层完全解耦，架构预留统一网关接口。v1.0 暂仅接入单一内置模型，多模型适配将在后续版本开放。

🧬

自我进化 Self-Evolving

GEPA 闭环设计使 Agent 在本地持续积累领域技能，重复任务效率随使用时长递增，无需云端训练。

📦

插件化分层 Layered Plugin

各层通过事件总线与接口契约通信，层内模块支持热插拔替换，降低变更成本与测试覆盖难度。

03 表现层 Presentation Layer

表现层是用户与 Nova Agent 的直接交互界面，负责解析多种输入形态、渲染流式输出、预览文件差异、管理终端状态徽章。

3.1 CLI 入口组件

组件职责输入形态
Command Parser 解析 argv、识别全局选项（-q, --batch）、子命令分发 nova -q "...", nova --batch *.md
REPL Engine 启动交互会话，维护终端状态机，处理 Tab 补全、历史召回、多轮上下文 nova 进入持续对话
Pipe Adapter 监听 stdin，将管道输入封装为标准消息注入 Session cat log | nova "..."
Batch Adapter 批量文件遍历，为每个文件创建独立子任务并聚合输出 nova --batch *.md

3.2 视觉反馈组件

组件职责关键技术
Stream Renderer 将模型流式输出分段渲染，支持 Markdown 实时解析、代码块语法高亮 Incremental Markdown Parser + ANSI 转义序列
Diff Previewer 文件修改前展示语法高亮 diff，支持 hunks 选择与确认 diff 算法（ patience / histogram ）+ 终端高亮
Progress Indicator 多步任务进度条、当前步骤说明、耗时统计，支持 Ctrl+C 优雅中断 终端进度条库 + 信号处理器
Status Badges 终端提示符集成 Git 分支、Python 环境、Agent 状态等上下文徽章 Shell 集成脚本（bash/zsh/fish hook）

3.3 Web Dashboard（v0.9 激活）

Dashboard 是表现层的可视化延伸，通过本地 HTTP 服务提供会话历史、任务状态与模型配置的图形化管理。所有资源本地打包，不依赖外部 CDN，数据不出本机。

组件职责关键技术
Dashboard Server nova dashboard 启动本地 Node.js HTTP 服务，提供 REST API 与静态资源托管 Express / Fastify + 本地 WebSocket（状态推送）
React 前端 会话列表、任务 DAG 可视化、模型参数面板、历史搜索与导出 React 18 + Vite 本地构建，零外部 CDN 依赖
State Sync Bridge CLI 与 Dashboard 间实时双向同步（任务进度、模型切换、会话状态），延迟 < 1s 工具控制面（HTTP-as-Tool）/ SQLite 变更监听 / 本地文件锁

3.4 消息网关（v1.0 激活）

消息网关将 Nova Agent 从纯 CLI 工具扩展为多端可用的智能助手，支持 Slack / Telegram / Discord Bot 接入，所有端共享同一 Agent 状态与记忆。

组件职责关键技术
Bot Adapter 统一适配各 IM 平台协议，将收到的消息转换为内部标准消息格式 Slack Bolt / node-telegram-bot-api / discord.js
Message Router 根据会话 ID 将消息路由到对应 Session Manager，维护多端状态一致性 本地消息队列 + 会话级锁
Cross-End Sync CLI 执行任务时，Bot 实时推送进度；Dashboard 同步更新；任一端切换模型全局生效 基于 SQLite 的发布/订阅 + WebSocket 广播

04 应用层 Application Layer

应用层是表现层与智能层之间的桥梁，负责会话生命周期管理、任务队列调度、权限分级控制与核心状态机维护。

4.1 会话管理器 Session Manager

  • ✓ 会话工厂 — 根据输入形态创建对应会话实例（REPL / Single / Pipe / Batch），分配唯一 Session ID
  • ✓ 上下文容器 — 管理对话历史、用户偏好、项目记忆的会话级缓存，支持跨会话记忆加载
  • ✓ 生命周期钩子 — 触发 session:start / session:end 事件，供 Hooks Engine 订阅
  • ✓ 持久化会话 — 异常退出后支持 nova resume 恢复未完成任务队列

4.2 任务调度器 Task Scheduler

  • ✓ DAG 执行引擎 — 接收 Task Planner 生成的有向无环图，拓扑排序后按依赖关系调度执行
  • ✓ 任务队列 — 支持内存队列（默认）与本地 SQLite 持久化队列（/goal 跨会话任务）
  • ✓ 状态机 — 每个任务经历 pending → running → paused → completed | failed → archived
  • ✓ 动态重调度 — 子任务失败时暂停 DAG，触发 Planner 重规划，更新依赖后继续

4.3 权限控制器 Permission Controller

  • ✓ 分级策略 — 默认三级：safe（自动执行）、normal（提示确认）、dangerous（强制确认 + 二次输入）
  • ✓ 规则引擎 — 基于命令模式、目标路径、Git 状态匹配分级规则，支持用户自定义 permissions.json
  • ✓ 审计日志 — 所有授权决策记录到本地审计日志，支持事后追溯

05 智能层 Intelligence Layer

智能层是 Nova Agent 的核心大脑，包含任务规划、子代理编排、GEPA 进化、上下文管理与模型网关五大核心组件。

5.1 任务规划器 Task Planner

任务规划器负责将高层自然语言需求拆解为可执行的子任务 DAG，并在执行过程中根据反馈动态调整。

[用户输入] ↓ Intent Classifier → 识别任务类型（代码/重构/分析/迁移） ↓ Context Collector → 拉取项目结构、相关文件、历史 Skill ↓ Sub-task Generator → 通过 Model Gateway 生成子任务列表 ↓ Dependency Resolver → 构建 DAG，拓扑排序，识别可并行节点 ↓ Scheduler → 提交到 Task Scheduler 执行队列
  • ✓ 意图分类器 — 基于规则 + 轻量模型快速识别任务类型，路由到对应 Specialist 模板
  • ✓ 上下文收集器 — 调用 Context Manager 的 RAG 与知识图谱，获取最相关的项目背景
  • ✓ 动态重规划 — 执行中遇到未预期错误时，冻结当前 DAG，重新生成后续子任务并合并

5.2 模型网关 Model Gateway

模型网关是智能层的底座，设计上屏蔽不同模型供应商的 API 差异，提供统一调用接口与上下文管理。v1.0 暂仅接入单一内置模型，架构层已预留扩展接口。

模块职责
Provider Registry 定义模型 Provider 注册接口（端点、认证、能力标签），当前仅激活默认 Provider，后续可扩展至 300+ 模型
Request Router 当前透传至默认模型；架构预留按任务类型自动选择最优模型与故障自动降级能力，待后续版本开放
Context Window Manager 监控上下文 token 消耗，触发 Context Manager 的压缩/摘要策略，防止超限
Response Adapter 统一不同模型的流式/非流式输出格式，标准化 tool_call / reasoning / content 字段

5.3 代码审查引擎（v0.11 激活）

代码审查引擎是智能层的质量守门员，负责静态分析当前 diff、识别潜在 Bug / 性能瓶颈 / 安全漏洞，并生成风格一致的跨文件重构方案。所有重构以 diff 形式预览，遵循"Plan-first + Diff 预览"哲学。

[Git Diff / Working Tree Changes] Diff Analyzer → 提取变更文件、行号范围、影响面 ↓ Static Linter → ESLint/Prettier/AST 规则扫描（本地执行） ↓ LLM Reviewer → 通过 Model Gateway 分析语义级缺陷（并发/安全/逻辑） ↓ Report Generator → 输出结构化报告：文件、行号、严重级别、修复建议 ↓ Refactoring Orchestrator → 符号重命名、组件提取、文档同步（生成 diff → 确认 → 执行）
  • ✓ Diff 分析器 — 解析 git diff 与工作区变更，识别变更的文件、函数、接口影响面，为后续审查提供精确上下文
  • ✓ 静态检查器 — 集成 ESLint、Prettier、TypeScript 编译器与自定义 AST 规则，在本地执行零延迟扫描
  • ✓ LLM 语义审查 — 将 diff + 项目记忆注入模型上下文，识别并发竞争、注入漏洞、逻辑死锁等深层缺陷
  • ✓ 重构编排器 — 执行跨文件符号重命名、组件提取、函数内联，自动更新所有引用（含字符串模板间接引用），并同步 JSDoc / README
  • ✓ 风格继承器 — 分析现有代码的 ESLint/Prettier 配置、引号风格、缩进与命名约定，确保生成的新代码与项目风格 100% 一致

06 核心机制 — GEPA 自进化循环

GEPA（Generate-Evaluate-Prune-Archive）是 Nova Agent 的核心差异化能力，使 Agent 能够在本地持续积累领域技能，重复任务效率随使用时长递增。

6.1 循环架构

[每完成 N 次工具调用] ↓ Generate → 自动生成 Skill Candidate 草案（任务模式 + 成功路径 + 失败教训） ↓ Evaluate → 对比历史执行数据，评估 Skill 的泛化能力与复用价值 ↓ Prune → 淘汰低价值/过时的 Skill，避免记忆膨胀与负迁移 ↓ Archive → 高价值 Skill 持久化为 Skill Document，存入本地 Skill Repository ↓ Context Manager → 后续任务自动加载匹配的 Skill 作为上下文前缀

6.2 核心组件

  • ✓ Skill Generator — 基于近期成功/失败的工具调用序列，提取可复用模式，生成包含任务描述、前置条件、执行步骤、预期输出的 Skill Candidate
  • ✓ Skill Evaluator — 在隔离测试环境中重放 Skill Candidate，对比基线执行耗时与成功率，量化 ROI 指标
  • ✓ Skill Pruner — 基于使用频率、时效性、任务漂移检测，定期清理失效 Skill，释放存储与上下文空间
  • ✓ Skill Archiver — 将验证通过的 Skill 序列化为结构化文档（Markdown + JSON Schema），存入本地 .nova/skills/，支持版本管理

6.3 与上下文管理的集成

GEPA 循环生成的 Skill Document 通过 Context Manager 的 RAG 系统编入向量索引。当用户发起新任务时，Context Manager 自动检索最相关的 Skill 并将其注入系统提示，实现"越用越聪明"的渐进增强效果。

07 核心机制 — 子代理编排

子代理编排器负责将复杂任务分解为可并行的 Specialist Agent，管理它们的生命周期、通信与结果聚合。

7.1 编排架构

[Task Planner DAG] ↓ Orchestrator → 识别可并行子任务，匹配 Specialist Agent 类型 ↓ Agent Pool → 复用空闲 Specialist 或 fork 新进程 ↓ (并行) Code Agent Test Agent Doc Agent Review Agent ↓ Message Bus → 中间结果交换，避免上下文冗余 ↓ Result Aggregator → 合并输出，冲突消解，统一结论 ↓ [统一输出 → 应用层]

7.2 核心组件

  • ✓ Agent Pool — 维护 Specialist Agent 进程池，支持复用、预热与资源上限控制，避免无限制 fork
  • ✓ Agent Factory — 根据任务类型动态加载 Agent 定义（代码、测试、文档、审查等），支持社区 Agent 市场安装
  • ✓ Message Bus — 基于工具控制面的消息总线，子代理间通过标准化的工具调用（HTTP-as-Tool）交换中间结果，所有网络交互均被封装为可审计、可拦截的工具行为
  • ✓ Result Aggregator — 对多代理输出进行冲突检测（如不同代理修改同一文件），生成合并策略（自动合并 / 用户仲裁）
  • ✓ Health Monitor — 监控子代理进程健康，异常崩溃时自动重启并恢复上下文

7.3 通信协议

子代理间采用标准 JSON Message 格式通信，消息类型包括：intermediate_result、dependency_request、conflict_alert、completion。所有消息通过 Message Bus 中转，Orchestrator 作为协调中心进行路由与日志记录。

08 核心机制 — 持久化记忆

持久化记忆系统负责跨会话保存项目结构、用户偏好与领域知识，并通过知识图谱构建实体关系，支撑精准上下文检索。

8.1 记忆分层

记忆类型存储内容存储介质检索方式
项目记忆 项目结构、技术栈、命名规范、模块依赖关系 SQLite + 本地文件索引 结构化查询 + 文件路径匹配
用户偏好 代码风格、常用命令、拒绝过的方案、模型偏好 SQLite + 本地 KV 存储 键值查询 + 会话自动注入
对话压缩 长会话中提取的关键决策、结论、用户意图摘要 SQLite 语义相似度检索
对话压缩 长会话中提取的关键决策、结论、用户意图摘要 SQLite 语义相似度检索 + 时间衰减排序
知识图谱 项目实体（模块、接口、数据库表）及其关系 本地嵌入式图数据库（如 SQLite + 内存图索引） 图遍历查询（类 Gremlin/Cypher）

8.2 知识图谱架构

[代码变更 / 用户输入] ↓ Entity Extractor → AST 解析 + 命名实体识别 ↓ Relation Builder → 建立模块依赖、调用链、数据流关系 ↓ Graph Store → 邻接表存储 + 增量更新 ↓ Query Engine → 支持路径查询、最近邻、子图匹配 ↓ Context Assembler → 将图查询结果注入模型上下文

8.3 上下文注入

  • ✓ @ 引用解析器 — 识别 @file:path、@pr:id、@doc:name，从对应存储加载内容并注入上下文
  • ✓ RAG 检索器 — 将用户输入编码为向量，从项目代码与文档的向量索引中召回最相关片段
  • ✓ 上下文压缩 — 长会话（50+ 轮）自动提取关键决策、结论与用户意图摘要，释放上下文窗口空间；压缩后仍可回答"之前我们为什么选方案 A"等历史追溯问题
  • ✓ 多模态解析 — 截图通过本地 OCR / 视觉模型提取结构化描述；URL 通过本地抓取 + HTML→Markdown 转换

09 扩展层 Extension Layer

扩展层通过 MCP、Hooks、Skill 三重体系，将 Nova Agent 从封闭工具转变为开放平台。

9.1 MCP 客户端 MCP Client

  • ✓ Server Discovery — 读取 mcp.json 配置，通过 stdio / SSE 连接 MCP Server，自动枚举工具列表
  • ✓ Tool Proxy — 将 MCP Server 暴露的工具封装为内部统一 Tool 接口，供 Task Planner 调用
  • ✓ Process Isolation — 每个 MCP Server 运行在独立子进程，崩溃不影响主 Agent，支持自动重启
  • ✓ Server Registry — 内置社区 Server 索引，支持 nova mcp add <name> 一键安装

9.2 Hooks 引擎 Hooks Engine

  • ✓ Event Bus — 全局事件总线，定义标准生命周期事件：before-command、after-file-change、on-error、session:start、task:complete 等
  • ✓ Condition Matcher — 基于正则、文件类型、Git 状态、命令模式等条件决定是否触发钩子
  • ✓ Script Executor — 支持 Shell 脚本、JavaScript 函数、HTTP Webhook 三种执行方式，沙箱内运行
  • ✓ Hook Registry — 读取 hooks.json 注册钩子，支持社区模板一键安装

9.3 Skill 仓库 Skill Repository

  • ✓ Local Store — 本地 .nova/skills/ 目录存储 Skill Document，支持 Git 版本管理
  • ✓ Market Client — 连接社区 Skill Market，支持 nova skills add <name> 下载预训练 Skill
  • ✓ Composition Engine — 支持将多个 Skill 组合为 Workflow，实现复杂领域任务自动化
  • ✓ Private Registry — 企业用户可配置私有 Git 仓库或本地路径作为 Skill 源，不与社区共享

9.4 市场体系（v1.0 激活）

市场体系是扩展层的社区化延伸，提供 Skill Market 与 MCP Server Market 两个核心频道，支持社区发布、评分、搜索与版本管理，使 Nova Agent 的能力边界持续扩展。

市场交付物核心能力

10 基础设施层 Infrastructure Layer

基础设施层为上层提供安全、可靠、可观测的底层能力，包括文件系统抽象、安全沙箱、记忆存储与进程管理。

10.1 文件系统抽象 FS Abstraction

  • ✓ Workspace Isolation — 所有文件操作限定在当前工作区，防止越界访问
  • ✓ Copy-on-Write — 文件修改前先写入临时副本，支持 /undo 回滚到任意历史状态
  • ✓ Git Bridge — 自动读取 Git 状态、分支、提交历史，作为上下文与决策依据
  • ✓ Watcher — 监听工作区文件变更，实时更新项目记忆与知识图谱

10.2 安全沙箱 Security Sandbox

Security Sandbox 的职责是运行时环境隔离，而非策略审查。所有策略判断（命令黑白名单、路径守卫、域名白名单）均由 Tool Controller 的统一策略引擎在调用前完成；Sandbox 仅负责在执行阶段提供进程级隔离与资源限制，作为最后一道防线。

  • ✓ 进程级隔离 — 每个 MCP Server、子 Agent、Hooks 脚本运行在独立进程，异常崩溃不影响主 Agent
  • ✓ 资源限制 — 对子进程设置 CPU、内存、执行超时上限，防止资源耗尽与 Fork 炸弹
  • ✓ 环境隔离 — 子进程继承最小环境变量集，敏感信息（SSH Key、API Key）默认不可见
  • ✓ Workspace 边界强制 — 文件系统操作通过 chroot / bind-mount 限定在工作区，防止越界访问（Path Guard 的底层强制）

10.3 记忆存储 Memory Store

  • ✓ Vector Store — 本地嵌入式向量数据库（如 sqlite-vec），存储代码与文档的 Embedding，支撑 RAG 检索
  • ✓ Graph Store — 内存图索引 + SQLite 持久化，存储知识图谱的节点与边
  • ✓ KV Store — 本地 SQLite 键值表，存储用户偏好、会话元数据、审计日志索引

10.4 工具控制面 Tool-as-Control-Plane

Nova Agent 采用"工具即控制面"（Tool-as-Control-Plane）的设计理念。 所有外部行为——文件读写、Shell 执行、HTTP 请求、进程通信、跨端状态同步——均被封装为标准化工具调用。 Tool Controller 不仅是路由层，更是所有工具的统一安全、策略与审计网关。 任何试图绕过 Tool Controller 直接操作文件系统或网络的代码都无法被执行，工具层即控制层。

[Agent / Dashboard / Bot / Hook] ↓ Tool Interface → 标准化工具调用（name + params + schema） ↓ Tool Controller + Policy Engine → 策略审查（命令/路径/域名/能力）→ 路由 → 审计 → 限速 → 拦截 ↓ HTTP-as-Tool / Shell Driver / FS Driver / MCP Client → 具体执行驱动 ↓ Security Sandbox → 运行时隔离（进程级 / 资源限制） ↓ Tool Response + Audit Log → 统一格式注入上下文，全链路可审计
  • ✓ 统一工具接口 — 所有外部交互（文件读写、Shell 执行、HTTP 请求、进程通信、MCP 调用）均实现同一套 Tool 接口（name / description / inputSchema / execute），模型无需关心底层实现差异
  • ✓ 统一策略引擎 — 为每个工具类型（shell_exec、file_read/write、http_request、mcp_call）注册独立策略规则：白名单、黑名单、需要确认、完全禁止。高危命令 AST 解析与路径守卫在此层统一完成
  • ✓ 分级授权 — Level 1 只读工具自动执行；Level 2 写操作需会话级确认；Level 3 高危操作逐条确认；Level 4 黑名单命中完全禁止。策略粒度可细化到工具名 + 参数条件
  • ✓ HTTP 工具化 — HTTP 请求不是裸 fetch，而是经过封装的 http_request 工具：自动附加域名白名单、速率限制、请求/响应日志、超时熔断。Dashboard、Bot、子 Agent 的通信均通过此工具完成
  • ✓ 可插拔传输 — 同一套 Tool 接口可由本地 HTTP Server、stdio 管道、SSE 流等不同传输层实现；更换传输层无需改动业务逻辑
  • ✓ 全链路审计原生 — 所有工具调用（命令、文件、网络、MCP）天然落入统一审计日志：谁、何时、通过哪个 caller、调用了什么工具、传了什么参数、返回了什么结果，结构化导出与回溯

11 核心数据流

11.1 典型请求生命周期

1. User Input Command Parser / REPL Engine → 解析意图，识别模式（Single / REPL / Pipe / Batch） 2. Session Init Session Manager → 创建/恢复会话，加载用户偏好与项目记忆 3. Context Assembly Context Manager → RAG 检索 + 知识图谱查询 + Skill 匹配 + @引用解析 4. Task Planning Task Planner + Model Gateway → 意图分类 → 子任务生成 → 依赖解析 → DAG 输出 5. Permission Check Permission Controller → 分级策略匹配 → 必要操作请求用户确认 6. Orchestration Agent Orchestrator → 创建 Specialist Agents → 并行执行 → Message Bus 通信 7. Extension Invocation MCP Client / Hooks Engine / Skill Repository → 工具调用 → 事件触发 → Skill 加载 8. Execution Tool Controller → FS Abstraction / Shell Driver / HTTP Driver / MCP Client → Security Sandbox → Tool Controller 先做策略审查（命令/路径/域名/能力），通过后进入具体 Driver 执行，执行过程受 Security Sandbox 运行时隔离保护 9. Result Aggregation Orchestrator Result Aggregator → 合并多代理输出 → 冲突消解 → 统一结论 10. Output & Archive Stream Renderer + GEPA Loop → 流式渲染 → Skill 评估归档 → 记忆更新

11.2 GEPA 循环数据流

Tool Call Logger → 记录每次工具调用的输入、输出、耗时、结果 ↓ (每 N 次调用触发) Generate Skill Generator → 提取成功模式，生成 Skill Candidate ↓ Evaluate Skill Evaluator → 重放验证，量化 ROI（耗时降低率、成功率提升） ↓ Prune Skill Pruner → 对比现有 Skill，淘汰低价值项 ↓ Archive Skill Archiver → 写入 .nova/skills/，更新向量索引 ↓ Context Manager → 将新 Skill 纳入后续任务 RAG 检索范围

11.3 跨端状态同步数据流（v1.0）

当消息网关激活后，CLI、Dashboard 与 Bot 共享同一本地状态存储。所有跨端通信不走裸 Socket，而是通过工具控制面（HTTP-as-Tool）完成：状态变更被封装为工具调用，经 Tool Controller 路由到对应端点。

[CLI 执行命令 / Agent 生成输出] Session Manager → 更新本地 SQLite 会话状态与任务进度 ↓ Tool Controller → 将状态变更封装为 sync_state 工具调用，路由到各端 ↓ (HTTP-as-Tool) Dashboard Server → 接收 sync_state 工具调用，WebSocket 推送至浏览器，实时刷新进度 < 1s Bot Adapter → 接收 sync_state 工具调用，推送进度消息至 Slack / Telegram / Discord ↓ [任一端切换模型 / 中断任务] 反向同步 → 同样通过 sync_state 工具调用写回 SQLite，确保全局状态一致
  • ✓ 工具即通道 — 所有跨端通信均为标准化工具调用，可被审计、拦截、重放；不存在裸 HTTP 或裸 Socket 直连
  • ✓ 本地优先 — 所有状态存储在本地 SQLite，不依赖云端，离线场景下多端同步仍可用
  • ✓ 延迟保证 — CLI → Dashboard 延迟 < 1s；CLI → Bot 延迟 < 2s（受 IM 平台 API 限制）
  • ✓ 冲突消解 — 同一时刻仅允许一个"写入端"，通过会话级文件锁避免状态冲突；Dashboard 与 Bot 默认只读，配置面板需申请锁

14 架构决策记录（ADR）

已确认 Accepted ADR-001 · 2026-05-21

本地优先架构，而非云原生

背景：AI Agent 类产品天然有云端部署的诱惑（便于集中训练、持续更新、多设备同步）。

决策：核心逻辑、文件操作、记忆存储全部放在本地 Node.js 进程，云端仅作为可选的模型 API 通道。

理由：

  • ✓ 数据隐私 — 用户代码、项目结构、审计日志不出本地，满足企业合规
  • ✓ 零延迟 — 文件操作和命令执行无需网络往返
  • ✓ 离线可用 — 本地模型（Ollama 等）可在无网络环境运行
  • ✓ 成本可控 — 无服务端运维成本，用户仅需承担模型 API 费用

代价：多设备同步需用户自行借助 Git / Syncthing；GEPA 循环的跨设备协同无法原生支持。

已确认 Accepted ADR-002 · 2026-05-21

插件化分层架构，而非微服务

背景：子代理编排天然适合微服务拆分，但 Nova Agent 是单机 CLI 工具。

决策：采用同进程分层 + 子进程 Specialist Agent 的混合模式，层间通过内存事件总线通信，Agent 间通过工具控制面（HTTP-as-Tool）通信。所有网络交互均被封装为标准化工具调用，而非裸 Socket 连接。

理由：

  • ✓ 启动速度 — 单进程 + 懒加载，避免微服务的冷启动延迟
  • ✓ 调试简单 — 本地单进程堆栈清晰，无需分布式追踪
  • ✓ 资源可控 — Node.js 单进程内存可控，子进程可设置资源上限
  • ✓ 部署简单 — 一个 npm 包即可安装，无需 Docker / K8s

代价：长期演进中若需扩展为服务端产品，需重构通信层为网络 RPC。

已确认 Accepted ADR-003 · 2026-05-21

GEPA 循环在本地执行，而非服务端训练

背景：云端集中训练可获取更大规模数据，但违背了本地优先原则。

决策：GEPA 的 Generate-Evaluate-Prune-Archive 全部在本地完成，Skill Document 存储在用户本地磁盘。

理由：

  • ✓ 隐私保护 — 用户的成功模式与失败教训是敏感信息，不上传云端
  • ✓ 即时反馈 — 本地归档的 Skill 可立即参与下一轮任务，无需同步延迟
  • ✓ 个性化强 — 每个用户的 Skill 仓库独立，不受社区平均水平的稀释
  • ✓ 零额外成本 — 无需云端训练集群与存储费用

代价：Skill 无法在设备间自动同步；新用户无法继承社区预训练 Skill（需通过 Skill Market 手动安装）。

已确认 Accepted ADR-004 · 2026-05-21

模型层完全解耦，不自研基础模型

背景：自研基础模型可打造差异化壁垒，但投入巨大且迭代慢。

决策：架构上通过 Model Gateway 统一适配 OpenAI-compatible API，不自研任何基础模型，专注 Agent 编排与工具链。v1.0 仅内置单一默认 Provider，多模型切换将在后续版本开放。

理由：

  • ✓ 零锁定（架构就绪） — 模型层已解耦，后续版本将支持在 Anthropic、OpenAI、本地模型间切换
  • ✓ 永远最优 — 模型技术迭代快，解耦后可随时接入最新 SOTA 模型
  • ✓ 专注优势 — 将资源投入 Agent 编排、GEPA、安全沙箱等差异化能力
  • ✓ 降低风险 — 避免模型训练的法律、合规、伦理风险

代价：模型 API 成本由用户承担；v1.0 暂不支持模型切换，模型能力上限由当前接入的 Provider 决定。

15 弹性架构与回退策略

本架构设计充分吸纳产品路线图中的风险预判，在关键高级能力（v0.12 及以后）的接入点上预留了清晰的回退路径。 任何高级功能若无法在验收标准内交付，均可无损收敛回单 Agent + 手动确认的核心模式，确保"高信任开发协作者"定位不被动摇。

风险 1：MCP 协议生态不成熟

影响：v0.7 的 MCP 支持可能因社区 Server 质量不稳定导致体验差。

架构回退：扩展层的 MCP Client 设计为可选插件，底层优先实现内置工具集（文件/Shell/Git）作为兜底。若 MCP 生态未成熟，可完全禁用 MCP Client 而不影响核心功能。

风险 2：多 Agent 编排复杂度超预期

影响：v0.12 的并行 Specialist Agent 可能导致上下文爆炸或结果冲突难以调和。

架构回退：智能层的 Agent Orchestrator 支持运行时降级为"伪并行"模式——在同一 Agent 进程内串行执行但快速切换角色提示（System Prompt），保留多角色体验但消除跨进程通信复杂度。该模式通过配置开关 --pseudo-parallel 激活。

风险 3：GEPA 自进化效果不明显

影响：v0.13 的 Skill 积累可能因评估指标设计不当，导致生成的 Skill 质量低、无法复用。

架构回退：GEPA 循环的 Generate 阶段可降级为"手动模板导入"：用户手动编写 Markdown 格式的最佳实践模板，Skill Repository 直接按模板执行，绕过自动评估与剪枝。保留结构化知识管理能力，去除自动进化的不确定性。

风险 4：Web Dashboard 与 CLI 状态同步难度

影响：v0.9 的 Dashboard 可能因 IPC 机制复杂导致状态不同步或延迟高。

架构回退：Dashboard Server 支持运行时降级为只读历史查看器：关闭 WebSocket 实时推送，改为定时轮询 SQLite 快照（5s 间隔），不实时同步正在执行的任务。该模式通过 --read-only 启动参数激活，显著降低架构复杂度。

总体回退原则

任何高级功能（v0.12 及以后）若无法在验收标准内交付，均收敛回单 Agent + 手动确认模式。 v1.0 的发布时间不因高级功能阻塞而无限延后。核心"高信任开发协作者"定位是最高优先级，所有扩展能力均围绕此定位生长，而非替代。

16 总结

Nova Agent CLI 架构以本地优先、安全优先、模型无关、开放扩展为设计原则， 采用五层插件化分层架构（表现层 → 应用层 → 智能层 → 扩展层 → 基础设施层）， 通过 GEPA 自进化循环、并行子代理编排、持久化记忆三大核心机制形成差异化竞争力。

关键接口已定义模块间契约，技术选型遵循"零额外依赖、零额外服务"的本地优先策略。 架构决策记录（ADR）明确了本地优先、单进程分层、本地 GEPA、模型解耦四大战略选择及其权衡。 弹性架构与回退策略章节为各 Phase 高级能力预留了清晰的降级路径，确保 v1.0 发布不受单一高级功能阻塞。

下一步行动：基于本架构设计与产品路线图，各模块按 Phase 渐进交付详细技术设计文档。 Phase 1（v0.1~v0.4）优先输出 CLI 入口、FS 抽象、Diff 引擎与 Shell 沙箱的状态机与错误处理策略； Phase 2~5 的扩展层与高级能力（Dashboard、消息网关、多 Agent 编排、GEPA）在对应迭代启动前输出接口定义与单元测试方案。
