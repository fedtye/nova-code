产品迭代路线图

以收敛版哲学为起点（高信任、单Agent、Human-in-the-loop），逐步向完整版愿景演进。 共 14 轮迭代，从最小可运行模型到全场景智能助手。

14 轮迭代 5 个阶段 功能驱动节奏

00 路线总览

迭代策略遵循"从内核到外围、从单Agent到多Agent、从基础能力到高级进化"的原则。 前 4 轮严格遵循收敛版哲学：所有关键操作可预览、可解释、可中断，不追求自治，专注执行质量。 从 v0.12 开始逐步引入完整版的高级能力（多Agent编排、GEPA自进化）。

1

Phase 1 — MVP核心

v0.1 ~ v0.4：能对话、能读文件、能编辑、能执行命令。所有修改需确认，所有操作可回滚。

2

Phase 2 — 能力扩展

v0.5 ~ v0.8：模型网关接口预留，支持管道模式与批量处理，接入MCP工具生态，Hooks生命周期扩展。

3

Phase 3 — 体验增强

v0.9 ~ v0.11：Web Dashboard可视化，持久化记忆跨会话，代码审查与重构能力。

4

Phase 4 — 高级智能

v0.12 ~ v0.13：并行多Agent编排，GEPA自进化循环，Skill系统积累领域能力。

5

Phase 5 — 产品成熟

v1.0：稳定化发布，消息网关全场景覆盖，企业版安全合规与私有化部署。

v0.1 核心骨架与对话 MVP

能运行、能对话的最小可执行程序。单一 Anthropic API 模型驱动，支持 REPL 持续会话与单行命令模式。

涉及架构层

表现层 智能层 基础设施层

前置依赖

无

交付物

  • ✓ CLI 命令解析 — 解析 nova、nova -q "..."、nova --version 等入口
  • ✓ 基础 REPL 引擎 — 进入后持续对话，内存级上下文记忆（最近 10 轮）
  • ✓ 模型网关（Anthropic） — 通过 Anthropic API 调用 Claude 模型，支持流式输出
  • ✓ 配置管理 — nova setup 引导配置 API Key，nova doctor 环境自检
  • ✓ 流式渲染（基础版） — 模型输出实时打印到终端，支持换行与代码块基础格式化

验收标准

  • ✓ API 连通性 — 成功调用 Anthropic API 并接收流式响应，首 token 延迟 < 2s
  • ✓ 上下文记忆 — REPL 中连续提问能引用前文，重启后上下文重置（符合 MVP 预期）
  • ✓ 单行模式 — nova -q "你好" 输出结果后立即退出，返回码 0
  • ✓ 安装体验 — curl -fsSL https://nova-agent.dev/install.sh | bash 一键安装成功

  • ✓ 任务调度器 — 状态机驱动，按 DAG 拓扑顺序执行，展示当前步骤 / 总步骤 / 预计耗时

R 风险与回退策略

每轮迭代均定义明确的前置依赖与验收标准，若某轮迭代受阻，可独立回退而不影响已完成阶段。

风险 1：MCP 协议生态不成熟

影响：v0.7 的 MCP 支持可能因社区 Server 质量不稳定导致体验差。

回退：优先实现内置工具集（文件/Shell/Git）作为兜底，MCP 作为可选扩展而非核心依赖。

风险 2：多 Agent 编排复杂度超预期

影响：v0.12 的并行 Specialist Agent 可能导致上下文爆炸或结果冲突难以调和。

回退：退化为"伪并行"——在同一 Agent 内串行执行但快速切换角色提示，保留用户体验但降低工程复杂度。

风险 3：GEPA 自进化效果不明显

影响：v0.13 的 Skill 积累可能因评估指标设计不当，导致生成的 Skill 质量低、无法复用。

回退：将 Skill 系统降级为"手动教程模板"，用户可手动编写最佳实践模板，Agent 按模板执行，保留结构化知识但去除自动进化。

风险 4：Web Dashboard 与 CLI 状态同步难度

影响：v0.9 的 Dashboard 可能因 IPC 机制复杂导致状态不同步或延迟高。

回退：Dashboard 改为只读历史查看器，不实时同步正在执行的任务，降低架构复杂度。

总体回退原则

任何高级功能（v0.12 及以后）若无法在验收标准内交付，均收敛回单 Agent + 手动确认模式，确保核心"高信任开发协作者"定位不被动摇。v1.0 的发布时间不因高级功能阻塞而无限延后。

S 总结

迭代哲学

本路线图以收敛版为坚实起点，确保前 4 轮迭代交付的每一个功能都经过"可预览、可解释、可中断、可回滚"的检验。 只有在核心能力稳固后，才逐步引入完整版的高级能力（多Agent、GEPA、消息网关）。 这种"先扎根、后生长"的策略，既避免了早期过度设计导致的稳定性风险，又为长期愿景保留了清晰的演进路径。

每轮迭代的详细技术设计（接口定义、状态机、算法逻辑、错误处理）将在迭代启动前单独输出， 技术栈统一采用 React + Node.js / TypeScript。
